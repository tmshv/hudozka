# Monorepo Migration — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Restructure hudozka into an npm-workspaces monorepo: move the Next app + writer + migrate script under `apps/`, extract shared UI / hooks / utils / text into `packages/` as `@hudozka/*`, keep `pb/` untouched.

**Architecture:** npm workspaces, no build step for packages (TypeScript source consumed directly via `"exports": { ".": "./src/index.ts" }`), single root `eslint.config.js`, single `tsconfig.base.json`. No Turborepo. Spec: `docs/specs/20260424-monorepo-design.md`.

**Tech Stack:** npm workspaces (npm 10+ / Node 24), Next.js 16, Vite 8 (writer), Node `--experimental-strip-types` (migrate), Vitest 4, ESLint 9 flat config, TypeScript 6.

---

## Conventions used in this plan

- **Repo root:** `/Users/tmshv/Workspace/__github_tmshv/hudozka`. All paths below are relative to this root unless absolute.
- **Working directory:** every `cd` command in this plan is idempotent — you can always `cd <repo-root>` at the start of a step.
- **macOS sed:** `sed -i ''` (BSD flavor). The user's environment is Darwin.
- **Commit messages:** one-line only (user rule). No body, no trailer, no co-author.
- **`git mv`** used for every relocation so history is preserved.
- **Verification gates:** after each phase there is a `Gate` step that must pass before starting the next phase.

### Task 3.5: Rewrite Dockerfile for workspace layout

**Files:**

- [x] **Step 1:** Replace contents with:

```dockerfile
FROM node:24.12.0-alpine AS base

# Install dependencies only when needed
FROM base AS deps
# Check https://github.com/nodejs/docker-node/tree/b4117f9333da4138b03a546ec926ef50a31506c3#nodealpine to understand why libc6-compat might be needed.

COPY package.json package-lock.json ./
COPY apps/hudozka/package.json ./apps/hudozka/
COPY apps/hudozka-writer/package.json ./apps/hudozka-writer/

COPY packages/ ./packages/

RUN npm ci
RUN npm install --os=linux --libc=musl --cpu=x64 sharp --workspace apps/hudozka

# Rebuild the source code only when needed
FROM base AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY --from=deps /app/apps/hudozka/node_modules ./apps/hudozka/node_modules
COPY . .

ENV NEXT_TELEMETRY_DISABLED=1

RUN npm run build -w apps/hudozka

# Production image, copy all the files and run next
FROM base AS runner
WORKDIR /app

ENV NODE_ENV=production
ENV NEXT_TELEMETRY_DISABLED=1

RUN addgroup --system --gid 1001 nodejs
RUN adduser --system --uid 1001 nextjs

COPY --from=builder --chown=nextjs:nodejs /app/apps/hudozka/public ./apps/hudozka/public

# Set the correct permission for prerender cache
RUN mkdir -p apps/hudozka/.next
RUN chown nextjs:nodejs apps/hudozka/.next

# Automatically leverage output traces to reduce image size
# https://nextjs.org/docs/advanced-features/output-file-tracing
COPY --from=builder --chown=nextjs:nodejs /app/apps/hudozka/.next/standalone ./
COPY --from=builder --chown=nextjs:nodejs /app/apps/hudozka/.next/static ./apps/hudozka/.next/static

USER nextjs

EXPOSE 3000
ENV PORT=3000

# server.js is created by next build from the standalone output
# https://nextjs.org/docs/pages/api-reference/next-config-js/output
ENV HOSTNAME="0.0.0.0"
WORKDIR /app/apps/hudozka
CMD ["node", "server.js"]
```

Key changes vs. pre-migration:
- Manifests for all workspaces are copied so `npm ci` can resolve the full workspace graph.
- `packages/` is copied wholesale because packages are consumed as TS source — their `package.json` + `src/` files are needed at install and build time.
- `sharp` reinstall scoped to `apps/hudozka` via `--workspace`.
- Build command: `npm run build -w apps/hudozka`.
- Standalone layout in Next 13+ for monorepos emits the server under `apps/hudozka/.next/standalone/apps/hudozka/server.js`, which is why the final `WORKDIR` is `/app/apps/hudozka` and the `COPY --from=builder ... /app/apps/hudozka/.next/standalone ./` lands the entire tree including `node_modules`.
- `public/` copied to `./apps/hudozka/public` so Next finds it at runtime relative to `WORKDIR`.

If Next's standalone trace doesn't produce `apps/hudozka/server.js` because of a monorepo detection issue, add `outputFileTracingRoot: path.join(__dirname, "../../")` to `apps/hudozka/next.config.ts` — see the verification step below.

- [x] **Step 2:** Add `outputFileTracingRoot` to `next.config.ts` to tell Next this is a monorepo.

Replace `apps/hudozka/next.config.ts` with:

```ts
import type { NextConfig } from "next"
import path from "node:path"
import { fileURLToPath } from "node:url"

const __dirname = path.dirname(fileURLToPath(import.meta.url))

const config: NextConfig = {
    output: "standalone",
    outputFileTracingRoot: path.join(__dirname, "../../"),
    trailingSlash: false,
    images: {
        remotePatterns: [
            { hostname: "hudozka.shlisselburg.org" },
            { hostname: "images.weserv.nl" },
            { hostname: "127.0.0.1" },
        ],
    },
}

export default config
```

Why: without `outputFileTracingRoot`, Next detects the workspace root incorrectly (finds a wrong `package.json`) and the standalone output ends up in a non-deterministic layout. Explicitly pointing it at the repo root makes the standalone tree predictable: `apps/hudozka/.next/standalone/apps/hudozka/server.js`.

Notes: the `import.meta.url`/`fileURLToPath` approach from the original draft caused Next's TS-config compiler to emit CJS (`exports = ...`) into a `.js` file inside an ESM package (`"type": "module"`), which fails at load time. Switched to `path.join(process.cwd(), "../../")` — `next build` runs with cwd = the app workspace directory, so this resolves to the repo root without touching ESM-only features. Verified: `npm run build` at root succeeds and `apps/hudozka/.next/standalone/apps/hudozka/server.js` is produced.
